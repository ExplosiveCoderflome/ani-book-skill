## 目标与边界

拆书的目标是把已授权来源转成可核对、可继续加工的创作知识，而不是复述原作、替代阅读或生成贴近原作的仿写稿。默认产物是分段笔记、八类分析小节、证据账本、覆盖地图和机制卡。

只分析以下来源：

- 用户提供、拥有或明确获准使用的 TXT、Markdown、正文片段或小说工作区；
- 无需绕过登录、付费墙、验证码、反爬限制即可访问的公开网页；
- 用户自己的稿件，此时使用诊断模式。

不要绕过访问控制，不要批量复制受保护正文，不要在公开仓库保存原文或包含大量原句的分析工作区。网页不可访问时，说明缺口并请用户提供文件。引用只保留支撑判断所需的短证据，并记录章节或页面来源。

## 选择目标、范围与深度

先确定一个主要目标：

- **结构学习**：故事发动机、阶段推进、冲突升级、高潮与回报。
- **人物学习**：角色功能、关系网络、成长弧、关键选择和形象变化。
- **写法学习**：视角、场景组织、对白、信息释放、节奏和断章。
- **商业学习**：点击驱动、读者承诺、爽点循环、付费拉力和流失风险。
- **稿件诊断**：检查用户作品的结构、人物、节奏、连续性和卖点。
- **综合拆书**：在预算允许时覆盖全部维度。

选择一种深度：

| 深度 | 默认范围 | 默认产物 |
| --- | --- | --- |
| 快速 | 用户指定范围，或有代表性的开篇与关键节点 | 覆盖声明、总览、主要优劣势、3-5 张机制卡 |
| 标准 | 可获得的完整范围，分段处理 | 总览、八类小节、证据账本、机制卡 |
| 深入 | 完整范围或明确的高质量抽样 | 标准产物，加阶段变化、反例检查、读者体验曲线和角色专题 |
| 诊断 | 用户自己的稿件范围 | 问题分级、证据、影响、最小修复和重检点 |

范围使用章节边界表达；没有可靠章节标题时使用稳定片段编号和字符区间。只有实际处理了全部范围，才可标记 `coverage: full`。其他情况一律标记 `coverage: sampled`，并列出未覆盖范围。

## 创建分析工作区

需要跨任务继续或分析长文本时，使用：

```text
analyses/<规范化名称>/
├── analysis-state.yaml
├── source-manifest.md
├── coverage-map.md
├── notes/
│   └── segment-001.md
├── sections/
│   ├── overview.md
│   ├── plot-structure.md
│   ├── timeline.md
│   ├── character-system.md
│   ├── worldbuilding.md
│   ├── themes.md
│   ├── style-technique.md
│   └── market-highlights.md
├── evidence-ledger.md
├── uncertainty-ledger.md
├── pattern-cards.md
├── graph/
│   ├── nodes.jsonl
│   └── edges.jsonl
├── retrieval/
│   └── analysis-index.sqlite3
└── recovery.md
```

默认不复制完整原文到分析工作区。`source-manifest.md` 记录来源类型、标题、作者或站点、路径或 URL、访问日期、来源版本、指纹、授权说明和实际分析范围。短期读取缓存不成为权威来源。

保持 `analysis-state.yaml` 简短：

```yaml
schema_version: 1
analysis:
  title: "待定"
  profile: "standard"
  status: "in_progress"
source:
  kind: "local_file"
  ref: "<path-or-url>"
  fingerprint: "sha256:<value>"
  coverage: "sampled"
  scope: "第1-50章"
artifacts:
  overview:
    path: "sections/overview.md"
    status: "missing"
next_action:
  type: "prepare_segment_notes"
  target: "segment-001"
```

使用 `missing`、`in_progress`、`ready`、`stale`、`blocked`。用户编辑过的分析产物默认受保护。来源指纹、版本或范围改变时，将依赖它的 notes、分析小节和机制卡标为 `stale`，不要静默重写。

## 运行分层拆解

### 1. 冻结来源

记录来源、权限、版本、指纹、范围和分析目标。网页来源记录具体章节 URL；同一 URL 内容变化时视为新版本。无法确认完整目录时，不声称覆盖整书。

### 2. 切分原文

优先识别标准中文章节标题、序章、番外和卷标题。章节边界不可靠时按约 6,000-16,000 字符切分，目标约 10,000 字符，并保留少量边界重叠以避免事件被截断。长书分批处理，不把全部原文一次塞入上下文。

每个 segment 使用稳定 ID，并记录：章节范围、字符范围、来源指纹、前后衔接和是否完整。

长篇来源完成首批 notes 后，按 [book-analysis-retrieval.md](book-analysis-retrieval.md) 建立轻量检索索引。关系追踪优先走 BookGraph，明确名称和事实优先使用元数据与全文检索，向量只做可选语义补召回。

### 3. 生成低推断笔记

每个 segment 只提取当前范围能支持的内容：

- 片段摘要与剧情节点；
- 时间事件和状态变化；
- 人物、关系、目标、行动与结果；
- 世界规则、势力、资源与限制；
- 主题、情绪和意象信号；
- 叙事技法、语言和场景组织信号；
- 读者满足点、钩子、回报和潜在流失点；
- 最多几条短证据及来源锚点。

不要在这一层断言整书主题、作者意图、隐藏动机或确定的商业结论。材料不足时留空，不要凑项。

### 4. 先生成总览

总览先固定一句话定位、题材、核心阅读承诺、目标读者、整体优势、整体短板和覆盖限制。后续小节使用它作为口径锚点，但仍必须由各自相关 notes 支撑。

### 5. 生成专项小节

| 小节 | 必须回答的问题 |
| --- | --- |
| 剧情结构 | 主线如何启动和重复运转；阶段如何升级；高潮、损失与回报怎样分布 |
| 时间线 | 关键事件顺序、阶段划分、角色状态变化和时间跨度风险 |
| 人物系统 | 主角发动机、配角与反派功能、关系网络、弧线和功能重叠 |
| 世界观 | 规则、资源、势力、舞台、限制，以及设定怎样实际推动剧情 |
| 主题表达 | 核心主题、情绪基调、母题及其通过行动而非口号出现的方式 |
| 文风与技法 | 视角、语言、描写、对白、节奏、信息释放、开章和断章方法 |
| 商业卖点 | 点击驱动、持续追读理由、爽点循环、读者匹配和商业风险 |
| 总览 | 上述结论的低风险汇总、覆盖声明和最值得继续深挖的问题 |

每个小节同时保存可读 Markdown 和紧凑的结构化结论。不要把大段分析重复复制到结构化层。

### 6. 做跨尺度综合

分别观察三个尺度：

- **章级**：目标、冲突、转折、回报、净变化和结尾拉力；
- **阶段/卷级**：目标切换、地图变化、对手升级、关系重组和阶段兑现；
- **全书级**：故事发动机、成长循环、揭示梯度、高潮序列和结局回收。

比较不同阶段的指标和模式，标出真正的变化点。不要把开篇特征直接外推为全书规律。

### 7. 生成机制卡

每张机制卡只沉淀可迁移原理：

```text
PATTERN-001｜机制名称
来源范围：第 X-Y 章
解决的问题：
运行机制：
必要前提：
读者回报：
失败方式：
反例或边界：
安全改造方向：
不可照搬元素：专名、独特设定、标志性桥段、原句表达
```

机制卡必须解释“为什么有效”和“换一个故事怎样重新设计”，不能只把原作桥段换名复刻。若用户要把机制用于新书，将它映射到 `novel-brief.md`、故事发动机、卷职责或章节合同，而不是直接生成仿写正文。

## 证据与不确定性合同

关键结论使用稳定 ID，例如 `CLAIM-001`，并记录：

- `kind`：`fact`、`inference` 或 `hypothesis`；
- `confidence`：`high`、`medium` 或 `low`；
- `scope`：适用章节或阶段；
- `source_refs`：章节、segment、URL 或字符区间；
- `evidence`：必要的短摘录或事实摘要；
- `counterevidence`：不符合该模式的章节或现象；
- `status`：`supported`、`contested`、`open` 或 `stale`。

事实只描述原文明确发生的内容。推断必须说明依据和适用范围。假设进入 `uncertainty-ledger.md`，不能伪装成事实。强结论优先要求两个独立来源锚点；只有一个锚点时缩小范围或降低置信度。

## 大体量作品的覆盖策略

预算不足以完整处理时，先建立抽样计划，而不是随机截取：

1. 开篇承诺与首个行动闭环；
2. 每卷开端、中点、高潮和卷末；
3. 主要人物首次进入、关系转折和退出节点；
4. 规则首次建立、首次被利用和首次付出代价的节点；
5. 全书关键揭示、低谷、终局和尾声；
6. 用户特别关注的范围。

`coverage-map.md` 必须区分已读、抽样、缺失和不可访问范围，并说明哪些结论只适用于样本。后续补充原文时，只增量生成新 notes，再重算受影响小节。

## 新增的质量检查

在交付前检查：

1. **覆盖诚实度**：报告是否明确区分全量与抽样。
2. **证据覆盖率**：关键结论是否有可定位依据。
3. **反例压力测试**：强规律是否检查过不符合它的章节。
4. **阶段漂移**：人物、节奏、文风或卖点是否在中后期发生变化。
5. **因果完整性**：是否说明机制的前提、行动、阻力、结果和回报，而不只是列标签。
6. **跨节一致性**：总览与人物、结构、市场小节是否冲突。
7. **可行动性**：机制卡是否能指导原创设计，同时保持足够改造距离。
8. **复述风险**：是否包含不必要的大段原文、过密情节复述或贴近原作的仿写指令。

## 诊断用户稿件

诊断模式沿用同一分段与证据链，但将每个问题写成：问题、证据、影响范围、严重度、最小修复、需要重检的下游产物。优先记录非阻塞质量债，再做局部修复；只有故事责任本身无法成立时才建议重规划。

未经授权不要修改原稿。用户批准修复后，先保护稳定正文和已成立内容，再按 [quality-and-repair.md](quality-and-repair.md) 执行。

## 恢复与交付

每完成一个 segment 或小节就保存结果并更新 `analysis-state.yaml`。单个小节失败时保留其他成功产物；恢复时只处理 `missing`、`stale` 或明确要求重做的部分。

最终报告必须包含：来源与授权边界、实际范围、覆盖类型、完成小节、关键结论、证据与不确定性、机制卡、未覆盖缺口和一个推荐下一步。
