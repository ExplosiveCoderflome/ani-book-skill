# 工作流路由

## 先判断用户里程碑

把请求归入一个主要里程碑，不要同时铺开整条生产链。

| 路由 | 常见输入 | 默认产物 | 默认停止点 |
| --- | --- | --- | --- |
| 创建新书 | 一句话灵感、题材、参考方向 | `novel-brief.md`，必要时补 `story-bible.md` | 用户可确认阅读承诺与故事发动机 |
| 卷规划 | 书级合约、预计篇幅、已有卷稿 | 卷战略、卷骨架或当前节奏板 | 当前卷职责与阶段回报可执行 |
| 章节生产 | 当前卷窗口、上一章、章节目标 | `plan.md`、`draft.md` | 本章正文可读且合同可验收 |
| 审查修复 | 计划或正文、已知问题 | `review.md` 和最小修改 | 阻塞问题解决或被准确升级 |
| 续写恢复 | `novel-state.yaml`、现有资产 | 下一项缺失或失效产物 | 工作流恢复到一个可继续状态 |
| 拆书分析 | 已授权文本、可访问网页、自己的稿件 | 分段笔记、分析小节、证据账本、模式卡 | 范围与覆盖度明确，关键结论可追溯 |
| 热门题材趋势 | 公开官方榜单元数据、榜单截图、表格或文本 | 榜单快照、趋势报告、3–5 张机会卡 | 榜单构成或可比日期变化明确，等待用户选择机会卡 |

## 创建新书

1. 先读取用户当前表达和已有 `novel-brief.md` 的确认记录；已确认、已授权 AI 决定或不适用的设置不得重复询问。
2. 按 [novel-brief.md](novel-brief.md) 的渐进确认协议，每轮只提出当前里程碑最重要的 2–3 个选择；每题提供 2–4 个内容选项，推荐项与影响说明清晰可见，并在整轮末尾统一提供自定义、全部接受、交给 AI 或跳过入口。
3. 按“开书定位 → 故事发动机 → 世界与角色 → 卷规划 → 首章前”逐步确认；不要提前抛出完整表单，也不要询问尚不影响当前产物的远期细节。
4. 再交付一句话 premise、阅读承诺、主角幻想和故事发动机，并注明哪些设定是用户确认、用户授权 AI 决定、AI 推荐待确认、待探索或不适用。
5. 未确认推荐只可用于对话预览；确认或授权前，不写入权威简报、不标记 `ready`、不生成正式下游资产。
6. 用户需求已经明确时，先用简短摘要复述已锁定设置，只询问真正阻塞当前里程碑的冲突或未决项。
7. 用户只要求局部创意时，不启动完整开书问答，也不自动生成正文。
8. 用户更改已确认的频道、阅读承诺、主角机制或叙事视角时，立即逐项使用 `stale` 状态报告失效产物，同时保护已有正文不被自动覆盖；频道或阅读承诺变更默认使未写卷战略、卷骨架、节奏板和章节计划失效，除非依赖检查证明某项不受影响。
9. 完全模糊的开书请求首轮固定只问读者频道、发布形态、题材或主要回报三项；不得追问主角、风格和篇幅，也不得先生成完整小说初案，最多附一条非权威的一句话示例。

## 卷规划

1. 先复用已确认的篇幅、阅读承诺和故事发动机；只确认仍未决的规划深度、首卷阶段回报或远期开放程度，且每轮不超过 3 项。
2. 区分战略、审查、骨架、节奏板和章节列表。
3. 上游战略改变后，将旧骨架和依赖它的未写规划标为 stale。
4. 保留已有正文和用户固定点。
5. 优先生成当前可执行窗口，避免远期过度规划。

## 章节生产

1. 生成第一章正式计划前，确认或取得 AI 决定权：叙事视角、节奏与语言倾向、单章长度和内容边界；每轮仍只询问最重要的 2–3 项。
2. 用户要求连续多章、提速、并行或子代理时，仍使用本页标准串行链；多子代理滚动预热已暂停，不得创建后续章节候选。
3. 缺章节计划时先生成计划与合同。
4. 已有可用计划时直接复用，除非用户给出新的修订要求。
5. 连续性已启用时，先创建本章上下文清单，再写正文。若使用跨书资产，Codex 先提出 `context-packages/chapter-XXX.assets.yaml`，校验通过后才将固定快照装入上下文；若章节涉及待审共享宇宙事件，只标记为需审查风险，不能自动采用正史更新。
6. 正文完成后先判断是否可读、是否兑现本章职责，再决定修复。
7. 只在正文稳定后写入章节差分、YAML 连续性（或旧账本）、角色状态、恢复记录与质量债；迁移后的工作区随后重建 Markdown 视图和 SQLite。预热候选不属于稳定资产。

## 审查修复

1. 把问题分成局部表达、章节义务、连续性、邻章分工和卷级结构。
2. 先修最小范围。
3. 保留已成立的爽点、转折、事实和角色声音。
4. 只有邻章职责失配才升级到重规划。

## 续写恢复

1. 读取状态索引并验证其中引用的文件是否存在。
2. 连续性已启用时，读取 recovery、相关 YAML 数据（或旧账本）、上一稳定章尾段、相关角色档案和当前卷窗口；SQLite 过期时直接降级为 YAML 读取。
3. 检查正文指纹；若稳定正文已变，将依赖的连续性资产标为 stale，不覆盖正文。
4. 找到第一个 missing、stale、blocked 或 in_progress 产物。
5. 检查其上游是否 ready。
6. 如果上游不可用，回退到最近一个可信里程碑，不猜测已完成事实。
7. 如果正文存在但状态落后，以正文和稳定资产事实为准，修正状态后继续。

## 拆书分析

1. 先确认来源、使用权限、分析目标和范围；线上来源不可访问时，请用户提供 TXT、Markdown 或正文片段。
2. 严格按 [book-analysis.md](book-analysis.md) 在 `analyses/<规范化名称>/` 创建来源清单、覆盖地图和 `analysis-state.yaml`；不得误用 `novels/`、`novel-state.yaml` 或另造冲突目录，也不要把未读取范围写成全书结论。
3. 先按章节或稳定长度分段生成低推断笔记；长篇来源完成首批 notes 后按 [book-analysis-retrieval.md](book-analysis-retrieval.md) 建立 BookGraph 与本地检索索引，再生成总览和所选专项小节。
4. 关键结论必须区分事实、推断和假设，并绑定章节或片段证据；强结论应检查反例。
5. 将可迁移经验写成机制卡，说明适用条件、读者回报、失效方式和改造方向，不提供贴近原作的仿写指令。
6. 用户稿件使用诊断模式，只生成问题定位和最小修复建议；未经授权不改正文。
7. 中断后保留已完成小节，只续跑缺失或 stale 产物。

## 热门题材趋势

1. “热门拆书、榜单题材、近期流行方向、赛道机会”进入榜单级趋势路线；“拆解某本书的人物弧、节奏或伏笔”进入书级拆解路线。
2. 用户未指定平台或读者频道时，先询问目标频道。默认每榜前 20 名，一次最多处理 3 个榜单。
3. 按 [hot-genre-trends.md](hot-genre-trends.md) 只读取公开官方榜单元数据；页面不可访问时接受用户提供的截图、表格或文本，不绕过任何访问限制。
4. 将快照保存到 `trends/<规范化范围>/`，从标题、标签和简介提取带证据与置信度的表层信号；不读取正文，不启用 BookGraph、RAG 或向量检索。
5. 单快照只描述“当前榜单构成”。至少两个同平台、同榜单、同统计周期的可比日期，才判断上升、下降、持续热门或新晋。
6. 生成包含限制声明的趋势报告和 3–5 张机会卡。只有用户明确选中的机会卡可以提炼进 `novel-brief.md`，且不得携带原始榜单、简介或其他机会卡。
7. 用户转向具体作品时停止趋势路线，明确升级到书级拆解并请求合法来源，不自动获取正文。

## 模糊请求的默认处理

- “帮我写小说”：先提供第一轮不超过 3 个高影响选择，至少处理读者频道倾向；可附一个推荐方向预览，但不直接固化或写几十章。
- “写起点男频玄幻，第三人称，约 200 万字”：确认摘要后复用这些决定，不重复询问；只处理当前故事发动机仍缺少的高影响项。
- “你决定，以后这些不用问我”：将相关设置记为当前项目的“用户授权 AI 决定”，直到用户撤回，不在后续里程碑重复询问。
- “我只想看三个灵感”：仅交付创意预览，不启动完整开书问答或创建工作区。
- “继续写”：先定位当前小说、最后稳定章节和下一章计划。
- “优化一下”：先说明优化对象和影响范围，再做最小修改。
- “全部自动完成”：按阶段持续推进，但仍保存中间资产和恢复点。
- “拆一下这本书”：先确认合法可用的来源、目标和范围；默认交付快速总览与覆盖度声明，不假装已经读完未提供内容。
- “最近热门玄幻”：进入热门题材趋势路线；若平台或男频/女频频道未指定，先询问目标频道，再读取榜单。
- “看看榜单里的某本书人物弧”：停止榜单趋势路线，升级到书级拆解，并先确认可用正文来源。
