# 工作流路由

## 先判断用户里程碑

把请求归入一个主要里程碑，不要同时铺开整条生产链。

| 路由 | 常见输入 | 默认产物 | 默认停止点 |
| --- | --- | --- | --- |
| 创建新书 | 一句话灵感、题材、参考方向 | `novel-brief.md`，必要时补 `story-bible.md` | 用户可确认阅读承诺与故事发动机 |
| 卷规划 | 书级合约、预计篇幅、已有卷稿 | 卷战略、卷骨架或当前节奏板 | 当前卷职责与阶段回报可执行 |
| 章节生产 | 当前卷窗口、上一章、章节目标 | `plan.md`、`draft.md` | 本章正文可读且合同可验收 |
| 审查修复 | 计划或正文、已知问题 | `review.md` 和最小修改 | 阻塞问题解决或被准确升级 |
| 续写恢复 | `novel-state.yaml`、现有资产 | 下一项缺失或失效产物 | 工作流恢复到一个可继续状态 |

## 创建新书

1. 先按 [novel-brief.md](novel-brief.md) 确认或推荐平台、读者频道倾向、题材、语气、叙事视角、节奏、情绪浓度、预计章节数、单章长度和更新节奏；不要要求新手一次填完所有项。
2. 再交付一句话 premise、阅读承诺、主角幻想和故事发动机，并注明哪些设定是用户确认、AI 推荐或仍待探索。
3. 再生成世界、角色和卷规划，不要用世界设定淹没故事钩子。
4. 用户只要求创意方案时，不自动生成正文。

## 卷规划

1. 区分战略、审查、骨架、节奏板和章节列表。
2. 上游战略改变后，将旧骨架和依赖它的未写规划标为 stale。
3. 保留已有正文和用户固定点。
4. 优先生成当前可执行窗口，避免远期过度规划。

## 章节生产

1. 缺章节计划时先生成计划与合同。
2. 已有可用计划时直接复用，除非用户给出新的修订要求。
3. 连续性已启用时，先创建本章上下文清单，再写正文。
4. 正文完成后先判断是否可读、是否兑现本章职责，再决定修复。
5. 只在正文稳定后写入章节差分、账本、角色状态、恢复记录与质量债。

## 审查修复

1. 把问题分成局部表达、章节义务、连续性、邻章分工和卷级结构。
2. 先修最小范围。
3. 保留已成立的爽点、转折、事实和角色声音。
4. 只有邻章职责失配才升级到重规划。

## 续写恢复

1. 读取状态索引并验证其中引用的文件是否存在。
2. 连续性已启用时，读取 recovery、相关账本、上一稳定章尾段、相关角色档案和当前卷窗口。
3. 检查正文指纹；若稳定正文已变，将依赖的连续性资产标为 stale，不覆盖正文。
4. 找到第一个 missing、stale、blocked 或 in_progress 产物。
5. 检查其上游是否 ready。
6. 如果上游不可用，回退到最近一个可信里程碑，不猜测已完成事实。
7. 如果正文存在但状态落后，以正文和稳定资产事实为准，修正状态后继续。

## 模糊请求的默认处理

- “帮我写小说”：先生成新书定位和钩子包，不直接写几十章。
- “继续写”：先定位当前小说、最后稳定章节和下一章计划。
- “优化一下”：先说明优化对象和影响范围，再做最小修改。
- “全部自动完成”：按阶段持续推进，但仍保存中间资产和恢复点。
