# 自动导演与恢复

## 目录

- 默认模式
- 里程碑
- 连续章节范围
- 停止条件
- 恢复顺序

## 默认模式

默认使用 `milestone_approval`：Codex负责推荐和执行，书级方向、故事发动机、卷战略及结构性重规划等待用户批准。用户明确说“你决定”或“全自动”时，可把对应范围记录为 `auto` 或 `delegated`，授权可以随时撤回。

导演状态只保存恢复所需投影：运行 ID、模式、状态、当前步骤、当前目标、批准的章节范围和停止原因。它不是对话历史，也不保存 Prompt 或正文。

## 里程碑

以下节点默认需要批准：

1. `novel_brief`：阅读承诺与故事发动机。
2. `story_bible`：长期对立、成长和结局方向。
3. `volume_strategy`：当前卷职责与阶段回报。
4. `stop_for_replan`：修改相邻章节或卷级分工。
5. 覆盖任何 `protected: true` 的资产。

批准只解锁下游，不把用户未确认的 AI 推荐伪装成用户决定。

## 连续章节范围

用户批准起止章节后，Codex可在该范围内逐章自动推进。每章必须完成连续性提交，下一章才可开始。到达范围末尾后状态转为 `waiting_approval`，请求新的范围或下一里程碑。

## 停止条件

只在以下情况停止全局链：

- 结构化判定为 `stop_for_replan`；
- 缺少无法安全推断的硬约束；
- 正文生成失败且没有可用内容；
- 即将覆盖用户保护资产；
- YAML、正文指纹或权威性出现完整性冲突；
- 用户主动暂停或撤销授权。

局部风格问题、一般质量债、可补写义务、审查暂时不可用、伏笔一般逾期均不得误停整书。

## 恢复顺序

1. 读取 `novel-state.yaml`，拒绝在旧 schema 上执行写操作。
2. 检查受保护文件、记录的指纹和缺失路径；发现用户编辑时先更新保护状态并使依赖产物 `stale`。
3. 连续性存在 YAML 权威源时验证 YAML；SQLite 缺失或过期时直接回退 YAML。
4. 优先恢复 `in_progress`，其次处理 `blocked`，再处理首个 `stale` 或 `missing`。
5. 如果状态和稳定正文冲突，以正文、用户编辑和已验收事实为准。
6. 恢复后只保留一个 `next_action`，说明目标、原因和是否需要批准。
