# 更新记录

本文件记录面向使用者的主要变化。开发中的内部调整不一定会单独记录。

### 2026-07-20

- 新增 Codex 原生 `novelctl.py` 工作区控制器，统一提供初始化、状态、唯一下一步、校验、显式迁移、步骤转换、阻断恢复、上下文、检查点、用量与稳定正文导出。
- 新工作区采用 `novel-state.yaml` schema v3；v1/v2 保持只读兼容，迁移必须显式执行并先生成备份，失败不会伪造 `ready`。
- 新增里程碑审批和章节范围授权投影；章节始终单 Agent 串行生产，每章稳定并提交连续性后才规划下一章。
- 开书三项选择成为 schema v3 的强制语义门槛；未确认频道、发布形态和主要阅读回报时，控制器不会启动书级简报。
- 新增用户资产指纹保护：已验收正文或规划被作者修改后保留原文，仅将未写下游标记为 `stale`。
- 新增 Codex-only 架构合同、生成步骤目录、质量债与恢复规则；不接入模型 Provider、Web UI、数据库权威或自研 Agent Runtime。
- 新增权威仓库与安装镜像的哈希校验/同步工具；同步不会删除镜像中的额外文件。
- `novelctl export` 仅导出正文和连续性状态均为 `ready` 的稳定章节，不再仅凭文件存在判断。
- 新增逐生成步骤 Token 用量合同；规划、正文、润色、审查、修复、连续性回灌、拆书和趋势报告等模型调用可分别记账。
- 新增 `token_usage.py`，支持追加记录、账本校验，以及按步骤和模型生成汇总视图。
- 精确 usage、兼容 tokenizer 估算和运行时不可获取事件严格分开；缓存输入与推理 Token 作为明细，不重复计入总数。
- 失败和重试保留独立事件；用量账本不得保存提示词正文、密钥或创作上下文，也不得进入章节上下文包。
- `novel-state.yaml` 只保存简短累计投影；追加式 JSONL 账本为用量事实源，汇总文件可随时重建。

### 2026-07-18

- 新增工作流宣传图，以可视化方式展示从选题、故事架构、正文审查到连续性回灌和下一章的完整创作闭环。
- 重写项目首页，围绕作者在长篇创作中的实际问题、可获得的工件和长期连续性价值展示核心能力，方便快速理解并开始使用。
- 将四阶段核心工作流前置为首页重点，清晰说明方向判断、故事发动机、单章生产和连续性回灌如何组成可持续创作闭环。

### 2026-07-17

- 新增参考小说拆解与用户稿件诊断路线，可处理已授权本地文本、可访问线上来源和现有小说工作区。
- 拆书采用“范围与指纹 → 分段笔记 → 总览 → 八类专项小节 → 证据与反例 → 机制卡”的可恢复流程，并明确区分全量覆盖与抽样分析。
- 关键判断区分事实、推断和假设，记录置信度、来源锚点、反例与适用阶段，降低长篇抽样外推和无依据结论风险。
- 新增阶段变化、读者体验、因果机制和原创改造检查，帮助作者学习有效原理，同时避免大段复述或贴近原作仿写。
- 新增私有 `analyses/` 工作区约定，原文和拆书产物默认不进入公开仓库。
- 长篇拆书新增轻量 BookGraph 和 SQLite 中文全文检索，可按人物、章节、维度、关系邻域和因果路径提取必要上下文，不再为每次专项分析重复读取全部 notes。
- 向量检索作为可选 sidecar 接入：未配置 embedding 时保持零第三方依赖并自动使用图、元数据与全文检索；配置兼容向量后可做余弦召回和排名融合。
- 新增独立的热门题材趋势路线，按需读取公开官方榜单元数据，并在网页不可访问时接受用户提供的截图、表格或文本。
- 新增私有 `trends/` 工作区、榜单快照合同、趋势报告和机会卡；原始榜单与未选机会卡不会进入小说创作上下文。
- 新增零第三方依赖的 `trend_snapshot.py`，支持校验快照、统计表层信号以及比较同平台同榜单的可比日期；脚本不内置爬虫、题材词典或趋势判断。
- 新增渐进式内容设定确认：在开书、故事发动机、世界角色、卷规划和首章前，每轮只引导确认 2–3 个高影响选择，并提供推荐与影响说明。
- 新增项目级选择记忆：已确认、已授权 AI 决定和不适用的设置不会被重复询问；未确认推荐只可预览，不能固化为权威资产或驱动正式下游规划。
- 连续性存储升级为 YAML 权威数据、自动生成的 Markdown 台账视图与可重建 SQLite 索引；可按章节和角色组装有限上下文，并通过检查点支持长期恢复。
- 新增旧工作区的安全迁移、YAML/索引一致性校验与直读降级：索引缺失或过期不会改变事实，也不会阻塞续写恢复。
- 章节生产明确保持单章串行回灌，暂停多子代理预热，避免未验收候选进入长期连续性。

- 提供贡献、安全与社区协作规范，方便社区共同维护工作流。
